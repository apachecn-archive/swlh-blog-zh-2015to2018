# 如何用 5 个简单的步骤规划一个工程项目

> 原文：<https://medium.com/swlh/how-to-plan-an-engineering-project-in-5-easy-steps-adc419a220d5>

或者“如何构建一个复杂的应用程序而不失去你永远热爱的头脑。”

![](img/b2114df41018f1e2055454ebeaaf2af9.png)

先说个免责声明:我不是项目策划专家。我不是一个认证的 scrum 大师，甚至也不是一个非常有条理的人。我只是一个长期编写代码和创建应用程序的人，我知道如果我计划得不好，我以后会为此付出代价。

如果你是一名工程师，你会看到它——这是一个必不可少的演示视频，向你展示如何使用一些闪亮的新语言/平台/SDK，在五分钟或更短的时间内构建一个博客/待办事项应用/wi-fi 支持的背部按摩烤箱。但问题来了。每一个演示都是从有人键入一堆代码开始的。在现实世界中，没有什么是从构建开始的，尤其是当期望的最终结果不仅仅是显示“Hello World”这几个字时

那么，究竟如何才能达到准备好直面代码的地步呢？嗯，有很多方法可以选择，但最终，它们都归结为仔细和彻底的规划。以下是我作为一名工程师采取的步骤，以确保我不会搞砸丢给我的项目。

# 1.理解真正的问题

不，说真的。

每个项目的存在都是为了满足一种需求，除非你理解这种需求，否则你无法制定一个成功的计划，更不用说构建一个应用程序来满足它了。在不了解项目目标的情况下继续前进就像闭着眼睛试着玩飞镖一样。你可能真的很棒，但是你不知道你的目标是什么。这听起来很基本，但它经常被忽视，所以我觉得有必要在这里指出来。告诉我你从未见过这种情况:

经理:我们需要一个用 COBOL 构建的微服务框架，它从 SOAP 服务中读取 XML，并将其翻译成 UTF-8 编码的斯瓦希里语，然后在一个全新的非 SQL 数据库中增加十六进制计数器，并使用非标准表情符号在我们的网站上显示总数。你能做到吗？

*工程师:*👍*【开始编码。]*

我们工程师倾向于从表面上接受规范文档。在默认模式下，我们通常只是低着头，打开音乐，开始构建一些东西，而不会问这样的问题:“为什么需要 COBOL 和斯瓦希里语？”

你知道吗？有时我们可以侥幸逃脱。有时候。如果项目很小而且我们很幸运。

但是很多时候，我们最终建造了错误的东西，或者更准确地说，我们最终建造了一个从技术上来说完全是我们被要求建造的东西，但这实际上并没有解决我们的利益相关者想要解决的问题。或者，我们甚至可以解决他们的问题，但却是以一种错综复杂的方式，尽管有一条简单得多的成功之路。

There’s got to be a better way…

因为事情是这样的:许多对非程序员来说听起来简单的特性实际上实现起来非常复杂，而对非程序员的麻瓜来说看起来像魔法的其他特性实际上是简单的构建，甚至是交钥匙解决方案。作为代码专家，我们的工作就是提供这种水平的洞察力——公开更好、更直接、更简单、更有效的方法来实现我们团队的目标。但是，如果我们不花时间问正确的问题，并确保我们真正理解我们的利益相关者在第一时间试图实现什么，我们就不知道寻找这些解决方案，我们经常会在一个月或一年后发现自己挖了一个不必要的复杂的洞，除了我们自己没有人可以责怪。

# 2.找出任何尚未做出的决定

在我多年的开发过程中，没有一次，我被交给一个大的需求规格说明书，它没有至少一个 *gotcha* 隐藏在范围中。总有一个妖精，一个没人想通的场景，躲在五段废话后面的阴影里，就等着跳出来毁了你的一天。帮你自己一个忙:在你深入构建和面临最后期限之前找到它。

And then, sometimes, it’s all goblins.

当你寻找这些未确定的问题时，记住魔鬼总是在细节中。例如，如果规范要求一个带有评论系统的博客，我几乎可以保证下面的一个或多个问题没有得到回答:

*   用户评论前必须认证吗？如果是这样，我们有没有考虑到这将如何发生？我们是否正在与任何社交 API 集成，以允许人们使用他们的脸书或 Twitter 账户进行身份验证？
*   个人意见在出现在公共网站上之前需要批准吗？如果是的话，这适用于所有的评论者，还是只是一些？谁可以批准评论？我们是否需要创建管理界面或管理用户角色来支持评论批准？
*   帖子公开前可以关闭评论吗？对于已经公开的帖子呢？如果一个帖子已经有公众评论，那么如果我们关闭评论会发生什么？现有的评论会消失吗，或者我们只是删除评论创建表单？
*   评论者可以回复现有评论吗？如果是，那么回复线程可以有多深？我们是否考虑了所有屏幕尺寸的设计中的嵌套注释？
*   我们是否有计划来处理评论和/或问题评论者中的不当/辱骂性语言？

这个清单可以一直列下去。你问的每一个问题都有可能挖掘出更多需要问的问题。我知道这很令人畏惧。跳过这一步，做出假设，并把一些东西带出门外是很诱人的，但是你知道当我们假设…坏事时会发生什么，伙计。坏事。因此，在开始构建之前，花时间认真考虑项目的需求，并找出哪些决策仍然需要做出。

**这一步的一些提示:**

1.  不要一次提出你所有的问题。你会让决策者不知所措、心烦意乱，通常，一个答案会适用于几个问题，或者使几个问题无效。因此，组织你的想法，并以一种让你的利益相关者感到被重视和被帮助的方式呈现出来。
2.  如果你发现了一个没有答案的问题，这个问题在得到答案之前会阻碍你的进步，让决策者知道。询问估计的响应时间，并要求他们遵守。这不是要做一个混蛋，而是要帮助团队按时完成任务。
3.  如果桌面上有两个或更多的选项，其中一个实现起来更简单或更复杂，那就把它表达出来。帮助每个人理解这些选项会对项目的时间表产生什么影响。技术难度并不总是拒绝更好选择的理由，但是成本可能会超过收益，您的决策者需要了解所有事实才能做出最佳选择。

# 3.将项目分成大的功能块

一旦你的大部分问题都得到了回答(因为说实话，你永远不可能提前得到*所有*问题的答案)，你就可以开始组织你的计划了。需要对数据库进行更改吗？创建一个名为“数据库变更”的文档，添加一两个要点来总结这些变更需要完成的内容，并在日程表上留出一些时间来思考细节(在步骤 4 中)。需要修改或创建哪些模块/类？对每个都做同样的事情。有没有依赖于其他团队的作品？记下它们，并将这些要求传达给受影响的团队或您的项目经理。对于你或你的团队需要构建的每一个部分，做一个需要完成的粗略的、高层次的大纲。有时这会让你带着新的问题回到第二步，这很好。让它们浮出水面，然后进入下一个功能部分。

一旦你觉得你的高层次计划处于一个好的位置，你可以考虑将它发送给其他人以获得反馈。到目前为止，你一直在检查交给你的计划工作。从这一步开始，你在做计划，所以从另一个工程师或股东那里获得理智检查，最好是在同一个项目中没有深入的人。一双全新的眼睛将能够看到你周围的树木，一起，你会发现你可能错过的东西。

Exactly.

# 4.将大的部分分解成小的部分，估计每个部分的工作量

对于较小或较简单的项目，您可以将此步骤与步骤 3 结合起来。对于更大、更复杂的项目，我建议分开处理。

为每一项任务列一个清单(检查两遍)(淘气或乖……好吧，我现在就退出),完成你刚刚列出的功能需要完成的任务。只要你觉得一切都包括在内，就不必按照执行的顺序列出它们。完成后，您应该会看到如下所示的内容:

*   创建一个数据库表来存储注释。应包括唯一的 ID、发表评论的日期/时间、被评论文章的 ID、评论者的用户 ID、被回复评论的 ID(如果适用)以及评论的文本内容。请务必遵循我们的组织标准和命名约定。*三十分钟。*
*   创建一个 comments 模型类来支持 comments 表。应该包括基本的 CRUD 功能和输入验证方法。 *1 小时。*
*   为每个模型方法编写单元测试，并测试它们。2 *小时。*
*   等等。

也许你用时间以外的术语来估计努力。那很好。我个人无法在任何其他单位思考，所以这就是我所做的。但目标是创建一个你(和你的经理)可以追踪的列表。当项目经理问你项目进展如何时，你可以说，“嗯，到目前为止，我已经完成了 X 项任务。我目前提前/落后于计划，并且我在项目中还有 X 小时/天/单位的工作量。”这比通常的“嗯，好吧，我想我可能已经走上正轨了”要有用得多。(你不是。)

再次，这是可能的(可能的？)您将发现更多本应在步骤 2 中提出的问题或本应在步骤 3 中概述的新功能。那很好。现在将它们浮出水面，冲洗并重复。

和以前一样，如果你有一个同事或朋友能够理解你的详细计划，并且真的喜欢你(或者只是一些你可以轻易接受的人)，让他们检查你的清单并给你反馈是个好主意。你计划得越好，你编写的代码就越好。

# 5.在构建之前编写文档

至此，你应该对项目有了一个真正扎实的理解，并有了一个可以执行的计划。如果你像我一样，你会渴望钻研代码。先别做！先写出一些粗略的文档。

Document first. Or your inner Gordon Ramsay will shame you.

现在，你写的文档类型会因组织和项目的不同而不同，但是要创造出*的东西*。您可能稍后会对其进行调整，但是这种策略遵循与测试驱动开发相同的哲学:确切地告诉您的代码它应该做什么，然后编写代码来完成它。如果没有别的，这将帮助你组织和编写更干净，更容易理解的代码，当你完成时更容易维护。

你可以写一份技术文档。你可以列出一系列的用例(如果还没有提供给你的话)。你可以设计一个用户流程图。但至少，你绝对应该写伪代码。

我想有些人可能会认为伪代码不是文档。我认为那些人错了。\_(ツ)_/写得很好的伪代码描述了你的应用程序的逻辑，如果你把它作为注释留在代码中(我通常这样做)，当未来的工程师修改你的代码时，它会帮助他们理解你的代码。这就是我把它包括在这里的原因。

伪代码的编写有不同的哲学和方式，我不想对风格作出规定。选择你自己的毒药。对我来说，重要的是我的伪代码(A)不包含实际的代码，并且(B)在项目构建时可以作为注释在我的代码中继续存在。我甚至不太担心这里的设计模式或最佳实践。我只是明白其中的逻辑。我以后总是可以重构的。一个伪代码的例子，我写它的方式，看起来像这样:

```
// Fetch all comments for this post from the database
// If we found one or more comments...
    // Create a data array to hold comment data
    // Loop through the comments, and with each comment...
        // Fetch the commenter's user data from cache/database
        // If we successfully retrieved the commenter...
            // Create a new object that contains:
                // the commenter's name
                // the commenter's avatar URL
                // the commenter's profile URL
                // the timestamp for the comment
                // the text of the comment
            // Add the comment to the comment data array
        // End If
    // End Loop
// End If
```

然后，如果你有同事或朋友愿意，在你执行之前，让另一个工程师检查你的逻辑是否有缺陷。

我知道。这可能会令人感到乏味和不必要，尤其是对于全新的开发人员和已经在 it 领域工作了足够长时间的工程师来说，他们认为自己可以即兴发挥。听一个编程经验丰富的人说，伪代码是 100%值得花时间的，尤其是当项目很大很复杂的时候。它允许您全面考虑您的应用程序逻辑，而不必同时担心语言语法、性能、代码风格和您环境的特点。

# 你的计划完成了。勇往直前去征服。

Remember what Miracle Max said: “You rush a code plan, you get rotten code.” Or something like that.

现在是*最后*写点代码的时候了！采取了前面的步骤后，您现在可以沉浸在荣耀中，因为您知道您不必在项目中途停下来问一个关于范围的问题，或者甚至弄清楚应用程序应该如何工作。就这么定了。你所要担心的就是写出你所能写出的最干净、最干燥、最易测试和最有效的代码。这本身就足够了，不是吗？想象一下，有了这种关注，你的代码审查会进行得更加顺利，你现在必须处理的真正“我构建错误”的误解和“bug”会减少多少。不客气

有没有其他适合你的方法？对于我可以在流程中做出的改进，您有什么建议吗？我很想听听！留下评论或回应，让我知道！
